WP8 File Format Analysis - UTV96_YawPowerCoils_Rev_0_4_Run_6.wp8

File is protobuf-encoded (NOT the simple binary format the old parser expects).

Structure:
  - Magic: FECE FACE (4 bytes)
  - Version byte: 0x01 (1 byte)
  - Protobuf message from offset 5

Top-level protobuf structure:
  field=1 (length-delimited): Channel block - repeated
    Sub-fields:
      field=1 (bytes): Metadata sub-message
        field=1 (varint): Some ID/flag (e.g., 102)
        field=3 (varint): Max value or flag (e.g., 65535)
        field=4 (varint): Some hash/ID
      field=2 (string): Channel name (e.g., "DCT Next Downshift Speed", "DCT Starter Inhibit")
      field=4 (bytes): Data point - repeated
        field=1 (float32): Value
        field=2 (varint): Timestamp (incrementing counter, likely milliseconds)

The old parser expected:
  - Fixed binary format with 00 10 channel markers and 03 10 row markers
  - Flat data rows with all channel values per row

The new file uses:
  - Protobuf encoding with varint field tags
  - COLUMNAR data: each channel block contains its OWN data points (field=4 repeated)
  - Each data point has: value (float32) + timestamp (varint)
  - Data is NOT row-oriented, it's column-oriented per channel

This is a newer version of the WP8 format (v2 protobuf) vs the older v1 binary format.
The parser needs to be rewritten to handle both formats.
